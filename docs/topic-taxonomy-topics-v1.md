## 0. 多轴如何展示（产品默认，不是打标默认）

**打标：** 一篇文可同时拥有轴 A（单选）+ 轴 B（单选）+ 轴 C（0–3）+ 轴 D（0–3）。  
**展示：默认不要把 A/B/C 一股脑铺开。** 存储齐全 ≠ 界面全显。

| 表面                   | 默认展现                                               | 理由                                                            |
| ---------------------- | ------------------------------------------------------ | --------------------------------------------------------------- |
| 时间线条目行           | **优先 C**（0–2 个）；空间够再加 **A** 一枚            | 用户扫列表时「关于什么」>「什么体裁」；B 在 AI 主导库里区分度低 |
| 条目详情顶栏           | **A + C**；B 仅在「非默认领域」时出现                  | 详情可稍密；B=「AI 与模型」时可隐藏以免噪音                     |
| 筛选 / 目录 / 高级过滤 | **A、B、C（及 D）分面可选**                            | 探索与收窄需要完整轴                                            |
| 主题目录卡片           | 按 **C**（及 D 实体卡、A 形态卡）组织，一般不按 B 铺墙 | B 太粗，不适合作主目录                                          |
| 订阅源默认             | 不展示条目级 B/C，除非源绑定了默认实体等               | 避免和源名重复                                                  |

### 优先级（争抢同一条 chip 位时）

1. **C 议题**（信息增益最高）
2. **A 形态**（改变阅读预期：教程 vs 快讯）
3. **D 实体**（若有，常与标题品牌重复，列表可省、详情可留）
4. **B 领域**（默认最弱；库内几乎全是 AI 时列表/详情可长期不展示，只进筛选器）

### 不推荐的默认

- ❌ 每条 news 固定打出「形态 + 领域 + 三个议题」五六个 chip
- ❌ 把 B 和 C 当成同一排无差别标签（用户会以为体系又扁回去了）

轴可以同时 **存在**；界面默认 **按场景抽显**。

## 1. 使用规则（打标与产品共用）

| 规则       | 说明                                                                    |
| ---------- | ----------------------------------------------------------------------- |
| 闭集       | 只能从下表 22 词中选；集外字符串一律丢弃                                |
| 每文上限   | **最多 3 个**；宁缺毋滥，置信不足则少标                                 |
| 空标合法   | 合集/无关/太泛时 topic 可为空（只靠形态/领域）                          |
| 主次       | 全文主线优先；顺带一句提到的议题不标                                    |
| 不与形态抢 | 「这是不是论文/教程/快讯」→ 形态轴，不进本表                            |
| 不与实体抢 | 「是不是 OpenAI/Claude」→ 实体轴，不进本表                              |
| 边界冲突   | 同时像两个词时，看 **文章在解决什么问题**，并参考各词「冲突时优先」     |
| 新增门槛   | 连续多周有稳定供给 + 现有词盖不住 + 有人愿意单独关注；季度评审可合并/删 |

### 合集（digest）策略

- 形态 = `合集` 时：议题主题 **最多 2 个**，只标整篇反复出现的主线；若无主线 → **topic 留空**。
- 不要把日报里每个小标题都标成 topic。

### 置信建议（给人/给模型同一套）

| 置信      | 行为                           |
| --------- | ------------------------------ |
| ≥ 0.75    | 可写入                         |
| 0.55–0.75 | 仅当没有更高置信的竞争词时写入 |
| < 0.55    | 丢弃                           |

## 3. 正反例表

说明：

- **正例**：应标该词（可与其它词组合时另注）
- **反例**：易误标；应改标它词或不应标本词
- **冲突时优先**：与近邻词打架时的默认选择

---

### 1. 大模型基础

**定义：** 讨论基座模型「是什么、怎么做出来、通用能力怎样」，不聚焦某一应用栈。

|                | 例子                                                                                                                                                                                            |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **正例**       | GPT/Claude/Gemini 等新基座发布且重点在模型能力与训练规模；Transformer 变体/架构论文；scaling laws；预训练方法综述                                                                               |
| **反例**       | 只讲 API 涨价与套餐 → `商业与产业`；只讲 ChatGPT 写代码功能 → `编码与开发`；只讲 o1 数学推理突破 → `推理能力`；只讲 7B 上手机 → `端侧与本地`                                                    |
| **冲突时优先** | vs 推理能力：文中核心是「推理专项突破」→ 推理能力；是「综合基座升级」→ 大模型基础。vs 开源生态：重点是权重开放/许可证 → 开源生态；重点是模型技术本身 → 大模型基础（可双标，技术为主则基础在前） |

---

### 2. 推理能力

**定义：** 复杂推理、思维链、测试时计算、数学/代码推理基准上的「想得更深」类能力（不是泛泛的「模型更强了」）。

|                | 例子                                                                                                                                                  |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| **正例**       | o 系列 / 深度思考模式解读；CoT、过程奖励、推理缩放；数学奥赛/逻辑基准上的推理 SOTA；「慢思考 vs 快思考」机制文                                        |
| **反例**       | 普通聊天体验变好但未谈推理 → `大模型基础` 或 `应用与落地`；Agent 多步干活但重点是工具与规划 → `Agent 智能体`；纯代码补全产品评测 → `编码与开发`       |
| **冲突时优先** | vs Agent：有环境交互/工具循环 → Agent；纯「脑子里多想几步」→ 推理能力。vs 评测与基准：文是报榜/比分方法 → 评测；文是推理方法本身 → 推理能力（可双标） |

---

### 3. Agent 智能体

**定义：** 模型作为主体做规划、调用工具、多步完成任务；含 agent 框架与协议实践（执行向）。

|                | 例子                                                                                                                                                                                                    |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **正例**       | Claude Code / 电脑操作 agent 产品与技术；多 agent 协作；ReAct/规划算法；「让模型自己跑完一个项目」类实践                                                                                                |
| **反例**       | 只介绍 MCP 协议规范、无代理行为 → `工具与 MCP`；单次 function call 教程式 → 更偏 `工具与 MCP`；纯推理模型发布 → `推理能力`；RPA 无 LLM → 不标或 `应用与落地`                                            |
| **冲突时优先** | vs 工具与 MCP：主题是「代理如何完成任务」→ Agent；主题是「怎么接工具/协议」→ 工具与 MCP。vs 编码与开发：Coding Agent 且主线是写代码产品 → 可双标，**编码产品形态强则编码在前，自主多步强则 Agent 在前** |

---

### 4. 编码与开发

**定义：** AI 辅助软件工程：补全、生成、评审、测试、IDE/CLI 编程工作流与代码向评测。

|                | 例子                                                                                                                                |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| **正例**       | Cursor / Copilot / Windsurf 产品更新；AI 写代码工作流；SWE-bench 等代码任务讨论；Vibe coding 方法论（重心在写代码）                 |
| **反例**       | 通用 Agent 操作浏览器下单 → `Agent 智能体`；模型开源但无代码任务 → `开源生态` / `大模型基础`；教「提示词技巧」与代码无关 → 不标本词 |
| **冲突时优先** | vs Agent：见上。vs 评测：文章只是报 SWE-bench 分数与榜 → `评测与基准`；讲如何让模型更会写代码 → `编码与开发`                        |

### 5. 多模态

**定义：** 图像/语音/视频等与文本的 **理解、对齐、融合**；强调跨模态能力，而非单一生成品类专题。

|                | 例子                                                                                                                        |
| -------------- | --------------------------------------------------------------------------------------------------------------------------- |
| **正例**       | 图文理解、截图问答、文档/UI 视觉理解；音视频理解通用模型；「一个模型吃所有模态」的统一架构                                  |
| **反例**       | 只谈 Midjourney / Flux 出图 → `图像生成`；只谈 Sora 类视频生成 → `视频生成`；只谈 TTS 音色 → `语音与音频`                   |
| **冲突时优先** | 生成模型但文中 70%+ 在讲理解/融合 → 多模态；品类就是「生图/生视频产品」→ 对应生成词。可双标：发布「既能看又能画」且两边都重 |

### 10. 端侧与本地

**定义：** 在手机、PC、边缘设备上本地跑模型；小模型、端侧芯片、离线推理体验与约束。

### 11. 开源生态

**定义：** 开源/开放权重、许可证、社区项目、开闭源策略与生态势力，**重心在「开放与社区」而非模型算法细节**。

| **冲突时优先** | vs 大模型基础：双标常见；**若标题/主线是「开源发布/许可/社区」→ 开源生态在前**                                         |

### 12. 数据与训练

**定义：** 数据集、合成数据、标注、预训练/微调/对齐训练 **方法与数据侧**（不是推理服务）。

|                | 例子                                                                                                                                                                        |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **正例**       | 新训练数据配比；合成数据管线；SFT/RLHF/DPO 训练配方；预训练算力与数据如何配                                                                                                 |

| **冲突时优先** | vs 安全与对齐：方法是训练目标/损失 → 可双标；主线是攻击与防御评测 → 安全与对齐                                                                                              |

### 13. 部署与推理工程

**定义：** 把模型 **稳定、低成本、高吞吐地跑在服务端/生产环境**：serving、量化、蒸馏用于部署、推理框架、GPU 集群成本。

|                | 例子                                                                                                            |
| -------------- | --------------------------------------------------------------------------------------------------------------- |

| **冲突时优先** | vs 端侧：见上。vs 开源生态：开源推理框架社区动态可双标，**工程实践主 → 部署**                                   |

|                | 例子                                                                                                                              |

| -------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| **正例**       | RAG 架构演进；向量库选型；分块/重排；「告别纯参数记忆」的知识注入实践                                                                  |
| **反例**       | 网页搜索产品整机（Perplexity 类）若主线是产品形态 → `应用与落地`，有明确 RAG 管线则双标；Agent 用搜索工具但主线是代理 → `Agent 智能体` |
| **冲突时优先** | vs 工具与 MCP：重点是检索质量/知识库 → RAG；重点是工具接口与协议 → 工具与 MCP                                                          |

---

### 16. 工具与 MCP

**定义：** 模型连接外部世界的 **接口与协议**：function calling、MCP、插件/工具标准、集成方式（不是业务场景本身）。

|                | 例子                                                                                       |
| -------------- | ------------------------------------------------------------------------------------------ |
| **正例**       | MCP 规范与生态；function calling 设计；工具调用协议对比；「给模型接数据库/浏览器」的接口层 |
| **反例**       | 用了工具的完整 agent 产品评测 → 主 `Agent 智能体`；企业微信客服落地 → `应用与落地`         |
| **冲突时优先** | vs Agent：见 Agent 条                                                                      |

---

### 17. 评测与基准

**定义：** 基准测试、排行榜、评测方法学、分数争议——**主线是「怎么量、量了多少」**。

|                | 例子                                                                                                                                           |
| -------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| **正例**       | 新 benchmark 发布；LMSYS/Chatbot Arena 排名变化解读；评测污染/刷榜争议；「这个榜不公平」方法文                                                 |
| **反例**       | 模型发布顺带贴几项分数 → 主 `大模型基础`（分数只是佐证，不标评测，除非文章结构就是评测专题）；单一产品体验文无方法 → `应用与落地` / 对应技术词 |
| **冲突时优先** | 专文评某一能力（如只评代码）→ `评测与基准` + 能力词（如 `编码与开发`）可双标                                                                   |

---

### 18. 应用与落地

**定义：** 垂直行业/场景中的真实使用与交付：教育、医疗、客服、办公自动化等 **「用起来」**，而非底座技术本身。

|                | 例子                                                                                                    |
| -------------- | ------------------------------------------------------------------------------------------------------- |
| **正例**       | 医院/银行 LLM 落地案例；销售/客服助手 ROI；行业解决方案复盘；「我们在生产环境踩的坑」                   |
| **反例**       | 新基座模型发布 → `大模型基础`；纯融资新闻 → `商业与产业`；教通用提示词 → 弱标或不标（除非场景绑定很强） |
| **冲突时优先** | vs 各技术词：文中 50%+ 是场景价值与实施 → 应用与落地；50%+ 是某项技术机制 → 技术词                      |

---

### 19. 商业与产业

**定义：** 钱、格局、竞争、商业化：融资、并购、定价、市场份额、战略动静（非技术深水）。

|                | 例子                                                                                    |
| -------------- | --------------------------------------------------------------------------------------- |
| **正例**       | 融资轮次与估值；大厂组织调整与 AI 战略；API 定价战；收购并购；「谁在赢市场」            |
| **反例**       | 技术报告 → 对应技术词；政策草案 → `监管与政策`；开源策略若从商业竞争写可双标 `开源生态` |
| **冲突时优先** | vs 监管：监管机关/立法 → 监管；公司市场行为 → 商业                                      |

---

### 20. 监管与政策

**定义：** 法律、政府、合规框架、出口管制、行业强制标准——**公共权力与规则** 向。

|                | 例子                                                                                                 |
| -------------- | ---------------------------------------------------------------------------------------------------- |
| **正例**       | AI 立法进展；芯片/模型出口管制；备案与合规指南；全球监管对比                                         |
| **反例**       | 公司安全团队红队技术文 → `安全与对齐`；「我们内部治理手册」无公共规则 → `安全与对齐` 或 `应用与落地` |
| **冲突时优先** | vs 安全与对齐：见安全条                                                                              |

---

### 21. 人物与访谈

**定义：** 以 **特定关键人物** 为阅读由头：访谈、对谈、圆桌、人物特写、大佬署名长文/演讲稿。主线是「听谁说 / 这个人怎么看」，不是某项技术说明书。

|                | 例子                                                                                                                                                                                                                                                                                                |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **正例**       | 对 Altman / Hassabis / 国内创始人的访谈；播客文字稿（嘉宾驱动）；「XX 谈 AGI」署名长文；人物特写                                                                                                                                                                                                    |
| **反例**       | 普通分析文顺带引用一句大佬 → 不标；产品发布会通稿无人物主线 → `商业与产业` 或技术词；纯政策文件解读 → `监管与政策`；技术深度文作者恰好有名 → 标技术词，不因作者有名就标本词                                                                                                                         |
| **冲突时优先** | vs 商业与产业：融资新闻里 CEO 说两句 → 商业；整篇是人物访谈顺带提融资 → 人物与访谈（可双标）。vs 形态「观点」：形态仍可标观点/分析；**本词回答「要不要当人物向专题追」**，两者可同时存在。vs 实体轴：实体标「是谁」；本词标「这篇是人物向内容」——访谈 OpenAI 老板：Entity=相关实体 + C=`人物与访谈` |

---

### 22. 创作与个人成长

**定义：** 主线是 **创作者怎么做 / 人怎么变**——写作与内容事业、播客与表达、职业转型、习惯与自我管理、成长复盘。不是模型技术说明书，也不是单纯融资新闻。

|                | 例子                                                                                                                                                                                                                                                     |
| -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **正例**       | 创作者 newsletter 谈转型与创作路径；收入结构/副业心态复盘；写作/播客方法论；个人成长实验与习惯；「辞掉全职后我怎样…」                                                                                                                                    |
| **反例**       | 纯模型发布/架构文 → 技术 C1；纯获客增长黑客且无个人叙事 → `应用与落地` 或 `商业与产业`；仅引用大佬金句无自身创作主线 → 不标或弱 `人物与访谈`                                                                                                             |
| **冲突时优先** | vs 人物与访谈：主线是「我的创作/成长」→ 本词；主线是「听某人说」→ 人物与访谈（可双标）。vs 商业与产业：个人收入心态/副业叙事 → 本词；公司融资与行业格局 → 商业。vs 形态合集：一期多段但整篇是创作者生活/成长拼盘 → A=合集 + C=本词（不要因合集就清空 C） |

## 4. 近邻易混速查

| 易混对                      | 一句话分开                         |
| --------------------------- | ---------------------------------- |
| 大模型基础 ↔ 推理能力       | 综合基座 vs 专项「想得更深」       |

| 数据与训练 ↔ 部署与推理工程 | 训 vs 推                           |
| 开源生态 ↔ 大模型基础       | 开放/社区 vs 模型技术              |

## 6. 明确不收录（v1 克制）

以下常见欲望词 **先不进闭集**，避免与轴或现词重叠：

| 不收录               | 原因                               |
| -------------------- | ---------------------------------- |
| AI（光杆）           | 过宽，无筛选价值 → 领域轴          |
| 提示词/Prompt        | 过碎；多数可归应用/编码/教程形态   |
| 创业                 | 与商业与产业重叠；单独拆词供给不稳 |
| 设计                 | 易与图像生成/产品混淆；有量再议    |
| 产品更新             | 属形态「发布」+ 实体，不是议题     |
| 搜索                 | 易与 RAG/应用糊；产品向归应用      |
| 科学智能/AI4S 等长尾 | 供给不稳，有量再加                 |

> 「人物与访谈」「创作与个人成长」已收入 C3，不再列于不收录。
