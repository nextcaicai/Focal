# Embedding 与搜索治理方案

> 状态：阶段 0–1 已完成；阶段 2 实现中（2026-07-12）  
> 范围：全库 embedding 带来的启动卡顿、使用变慢、搜索结果过多/过乱  
> 关联文档：`docs/startup-search-performance-governance.md`（启动分层 P0/P1）  
> 关联实现：`entry-embedding/backlog.ts`、`entry-enrichment/trigger.ts`、`library-search.ts`

---

## 1. 核心决策（Product）

| 决策             | 结论                           | 对用户的影响                                                 |
| ---------------- | ------------------------------ | ------------------------------------------------------------ |
| 全库 embedding   | **不要**（不作为默认基础设施） | 启动更快、后台更安静、内存更小                               |
| 关键词全库搜索   | **要**（默认搜索路径）         | 不依赖向量，全库标题 + RSS 摘要始终可搜                      |
| 入库即已读 embed | **不要**                       | 与 BYOK 政策一致，避免为「用户不会看的旧文」付 indexing 成本 |

### 1.1 一句话战略

**搜索默认 = 关键词全库；embedding = 只服务「用户会碰到的文章」的语义增强，按需启用。**

### 1.2 与现有 ingest 政策对齐

初次订阅（`local-rss/service.ts`）已有分层：

- 最新 N 篇（`INITIAL_SUBSCRIPTION_UNREAD_COUNT`）→ 未读 + BYOK enrich
- 其余 RSS 窗口 → **入库即已读**，不走 BYOK

Embedding 应对齐同一逻辑：

- **Embed**：未读、用户真正打开/读过的、收藏（可选）
- **不 embed**：初次订阅标为已读的历史、`historyBackfill` 导入的旧文
- **按需 embed**：搜索命中但未索引、用户手动标未读（阶段 3，可选）

### 1.3 当前产品范围说明

- **已上线**：侧边栏库搜索（关键词 + 可选语义）
- **未作为决策依据**：「我的话题」等未开放能力，不得作为「必须全库 embed 已读历史」的理由

---

## 2. 问题根因

全库 embedding 被当成启动与搜索的前置条件，导致：

1. **启动**：~4000 条 embedding JSON 全量 hydrate（~15s，与 entry 正文同级瓶颈）
2. **日常使用**：后台持续 backfill / upsert，占 CPU、API、主线程
3. **搜索**：每次 query 调 embedding API；语义扫描 + 弱相关结果泛滥
4. **质量**：摘要子串命中即收录（50 分）；RSS 摘要与列表 AI 摘要不一致，用户困惑

关键词路径**不扫正文**，已读历史**关键词仍可全库搜索**——不必为搜索默认可用性维护全库向量。

---

## 3. 目标

| 指标                      | 现在（典型 3k 库）      | 目标                          |
| ------------------------- | ----------------------- | ----------------------------- |
| 首屏可交互                | ~17s                    | **2–3s**（P0 分层 hydration） |
| 实体词搜索（codex、华为） | 等 query embed + 结果多 | **秒出、结果可控**            |
| 默认 embed 范围           | 全库 read + unread      | **未读 + 用户交互过的**       |
| 入库即已读 backfill       | 全量排队                | **默认跳过**                  |

---

## 4. 实施阶段（单主线，按序执行）

```
阶段 0 止血 → 阶段 1 减负 → 阶段 2 搜得准 → 阶段 3 打磨（按需）
```

### 阶段 0 — 止血（✅ 已完成）

| 项                   | 状态 | 改动方向                                              | 文件                                              |
| -------------------- | ---- | ----------------------------------------------------- | ------------------------------------------------- |
| P0 分层 hydration    | ✅   | Phase 1 挡 `appIsReady`；embedding 进 Phase 2         | `hydrate.ts`、`hydrate-deferred.ts`               |
| 实体词跳过 embedding | ✅   | `isEntityLookupQuery` → 不调 query embed、不跑 cosine | `library-search.ts`、`useQueryEmbeddingVector.ts` |
| 结果 cap             | ✅   | 总命中 ≤80，展示 ≤50                                  | `library-search.ts`                               |
| 纯语义上限           | ✅   | 关键词分为 0 的命中 ≤20                               | `library-search.ts`                               |
| 搜索快照隔离         | ✅   | 同 query 期间 embedding upsert 不重算                 | `library-search.ts`（P1）                         |

**阶段 0 完成标准**：打开可交互；搜 `codex` 明显更快；列表不再刷几百条。

### 阶段 1 — 减负（✅ 已完成）

| 项                              | 状态 | 改动方向                            | 文件                            |
| ------------------------------- | ---- | ----------------------------------- | ------------------------------- |
| `entryNeedsEmbedding` 尊重 read | ✅   | 入库即已读 → 默认不索引             | `backlog.ts`                    |
| backfill 与 BYOK 对齐           | ✅   | embedding backfill 只传 `unreadIds` | `entry-enrichment/trigger.ts`   |
| 停止无差别全库补洞              | ✅   | `enqueueAllMissing` 仅未读缺口      | `job-service.ts`                |
| UI 文案                         | ✅   | 设置页 + 搜索头改为「未读索引」     | `locales/ai/*`、`locales/app/*` |

**阶段 1 完成标准**：backfill 队列与 API 压力大幅下降；Phase 2 hydrate 数据量显著减少。

**不删除已有向量**；仅停止对新入库即已读条目的默认 indexing。

### 阶段 2 — 搜得准（进行中）

| 项                       | 状态 | 改动方向                                                  |
| ------------------------ | ---- | --------------------------------------------------------- |
| 零关键词 → 不跑语义      | ✅   | 仅 `keywordHitCount > 0` 时语义重排                       |
| 实体词摘要命中 cap       | ✅   | description-only ≤20（`applyEntityLookupDescriptionCap`） |
| 设置页 / 搜索头文案      | ✅   | 未读索引；全库关键词始终可用                              |
| 搜索与展示对齐           | ⏳   | 标「摘要命中」或纳入 AI 摘要                              |
| （可选）embedding 源文本 | ⏳   | title+description only；渐进 re-embed                     |

### 阶段 3 — 按需（库规模或 profiling 触发）

- 搜索命中 lazy embed
- 向量二进制存储
- sqlite-vec / Web Worker（库 > 1 万）

## 5. 搜索行为规格（决策后）

### 5.1 默认路径：关键词全库

- 字段：标题、RSS `description`、译文标题/摘要
- **不扫 HTML 正文**
- 全库所有 entry（含入库即已读）均可命中

### 5.2 增强路径：语义（可选）

- 前置条件：用户开启 Settings > AI > Embedding
- 范围：**仅已 embed 的条目**（阶段 1 后 ≈ 未读 + 用户交互过的）
- 实体型 query（≤8 字等）：**关键词独占**，不调 query embedding
- 长句 / 意译：仅当有关键词命中时做语义重排；零命中 = 空结果

### 5.3 明确非目标（本阶段）

- 不为「全库语义覆盖」维护全库向量
- 不为未开放功能（如 My Topics）预支全库 indexing 成本

---

## 6. 验证

### 6.1 启动

1. 完全退出 → 重开 → 帮助 → 打开日志文件
2. Phase 1 `hydrate total` < 3000ms
3. `[perf] hydrate.deferred` 中 `entryEmbedding` 耗时下降（阶段 1 后向量条数减少）

### 6.2 搜索

1. 搜 `codex`：无 query embed 等待（阶段 0）；结果条数 ≤ cap
2. 搜长句意译：语义仅作用于已 embed 子集
3. 入库即已读旧文：关键词可命中；未 embed 时不应纯语义入选

### 6.3 embedding 队列

1. 新订阅 feed：仅未读条进入 embedding backlog
2. 历史已读条目不持续入队

## 8. 与 v0.2.6 changelog 的关系

v0.2.6 曾写「语义索引覆盖已读历史」。本决策**收回该默认策略**，改为：

- 产品承诺：**全库关键词搜索**
- 语义索引：**活跃条目子集**，非全库标配
