# 订阅语料与入库策略 — P0 设计

> 状态：已确认并实现（2026-07-10）  
> 范围：订阅源入库 / 未读 / AI 增强策略；**不包含搜索模块**  
> 关联实现：`apps/desktop/layer/renderer/src/modules/local-rss/service.ts`、`refresh-scheduler.ts`  
> 后续：搜索与 topic 闭环依赖本 P0 把本地语料做厚（另文）

---

## 1. 背景与问题

Focal 是本地优先的 RSS 阅读器。搜索、Recommended、topic 等都依赖**本地已入库的条目**。

当前新订阅逻辑（`INITIAL_SUBSCRIPTION_UNREAD_COUNT = 5`）把三件事绑在同一个数字上：

1. **只把最新 5 条写入本地 DB**（更老的直接丢弃）
2. **写入 `historicalCutoff`**，之后刷新继续丢弃 cutoff 之前的条目
3. **AI 增强只跑这 5 条**

效果：

| 维度                    | 结果                               |
| ----------------------- | ---------------------------------- |
| 时间线                  | 不会被历史 backlog 淹没 ✅         |
| BYOK token              | 新订不会爆 ✅                      |
| 本地可检索 / 可回顾语料 | 每源极薄 ❌                        |
| 老订阅                  | 永久只有「当年 5 条 + 订后增量」❌ |

RSS 日常刷新**不会**自动补回当初丢掉的历史：它只拉当前 feed 窗口，且 cutoff 会继续挡掉窗口内更早条目。

---

## 2. 目标

在 **不淹时间线、不爆 BYOK** 的前提下：

1. 每个订阅源在本地拥有 **RSS 当前窗口能提供的完整条目集**（检索与回顾的地基）
2. **未读与 AI 消耗**仍可控、可预期
3. **已订阅源**在升级后自动补齐「窗口内仍可拿到、但本地缺失」的历史
4. 为后续搜索 / topic 铺路，但 **本阶段不实现搜索**

### 非目标

- 搜索 UI、FTS、语义搜、Follow as topic
- 站点多年全量归档（RSS 窗口外）
- 订阅时或迁移时全库跑 LLM / 全库 embedding backfill
- 修改默认未读条数 N（本阶段锁 5）
- 对 Ingest 条数设 soft cap

---

## 3. 核心原则：三策分离

| 策略                  | 管什么                | 默认决策                                        |
| --------------------- | --------------------- | ----------------------------------------------- |
| **Ingest depth**      | 本地库有多少          | 当前 RSS 窗口 **全量入库，不封顶**              |
| **Unread policy**     | 多少算未读 / 进时间线 | 最新 **N = 5** 未读                             |
| **Enrichment policy** | 谁花 BYOK token       | 默认仅 **新增量 + 未读窗口**；历史已读不自动 AI |

一句话：

> **历史可以进库，默认已读、默认不 AI。**  
> 入库服务「语料厚度」；未读服务「注意力」；增强服务「正在被消费的信息」。

## 4. 已确认参数

| 参数             | 决策                                                      |
| ---------------- | --------------------------------------------------------- |
| 本阶段范围       | 只做订阅源语料与策略，**不做搜索**                        |
| 未读 N           | **5**（后续可再调）                                       |
| Ingest cap       | **无**；吃满当前 feed 返回的全部条目                      |
| 老订阅迁移       | **升级后默认补历史**                                      |
| 迁移体验         | **A：静默后台补，结束后轻提示**（不弹确认页）             |
| Embedding 存量   | **P0 不补**；仅新内容 / 未读窗口在开启 embedding 时自动算 |
| 打开文章 Summary | **按钮触发**，不因「打开」自动生成                        |

## 5. 行为规格

### 5.1 新订阅

```text
Fetch 当前 feed 全部条目（不封顶）
  → 全部写入本地 DB
  → 最新 5 条：unread
       → 若用户开启对应开关：可跑默认 AI 流水线
         （summary / translation / tags / qualityScore / embedding）
  → 其余：read
       → 不造成时间线洪水
       → 不自动 AI、不自动 embedding
```

入库字段（检索与阅读底座）：

- 标题、作者、链接、发布时间、feed 关联
- RSS 自带 `description` / `content`（有则存）
- 不默认写入 summary / tags / quality / embedding（除非落入上面的未读/增量自动流水线）

### 5.2 Cutoff 语义

**目的（保留）：** 防止已处理过的旧条目在刷新时再次变成未读或造成噪音。

**错误用法（废除）：** 用 cutoff 阻止「feed 窗口内、本地缺失」的条目入库。

**目标行为：**

- 以 **entry id 去重 + 已读状态** 为主
- 允许窗口内缺失条目在迁移 / 手动导入 / 策略修正后进入本地
- 补入的历史条目 **一律已读**，且 **不** 自动 enrich

实现时需重写或修正现有 `historicalCutoff`（`local-rss:history-cutoff:*`）与 `refreshLocalRssFeed` 中 `isInitialSubscription` 分支，使「只存 5 条」变为「全量存、只 5 未读」。

### 5.3 日常刷新

| 情况                             | 行为                                              |
| -------------------------------- | ------------------------------------------------- |
| 真正的新条目（本地未见）         | 入库；按产品规则标记未读；可走默认 AI（若开关开） |
| 已在库中                         | 合并状态，不重复刷未读                            |
| 窗口内本地缺失（迁移后不应常见） | 可入库为已读；不自动 AI                           |
| 指望「多刷几次历史自己回来」     | **不会**；靠迁移 / 手动导入                       |

### 5.4 老订阅升级迁移（默认、静默）

**触发：** 客户端升级到包含本能力的版本后，对每个已订阅本地 RSS 源执行一次。

**行为：**

1. 再 fetch 该源当前 feed
2. 将窗口内、本地尚不存在的条目写入 DB
3. 补入条目全部标为 **已读**
4. **不** enqueue summary / translation / tags / qualityScore / embedding
5. 修正与本策略冲突的 cutoff，避免补完又被刷掉
6. 全程 **静默后台**；全部完成或本轮结束后 **轻提示** 一次  
   文案方向：「已为现有订阅补全当前可获取条目（不消耗 AI）」  
   （具体 i18n key 实现时定；`en` / `zh-CN` / `ja`）

**约束：**

- 不阻塞阅读主路径（后台队列 / 空闲调度均可）
- 单源失败可跳过并记日志，支持后续重试；不因单源失败中断全部
- 用持久化 flag（如 `local-rss:history-backfill-v1-done` 或每源 done 标记）保证 **默认只自动跑一轮**
- 网络差或源很多时允许分批，避免卡死 UI

### 5.5 手动「导入可获取历史」（建议 P0 一并做）

与迁移 **同一套逻辑**，入口在源级设置（或源菜单）：

- 名称方向：Import available history / 导入当前可获取条目
- 已读、无 AI
- 用于：迁移失败重跑、用户想再拉一次窗口

文案必须诚实：**当前 feed 能返回的条目**，不承诺「全部历史 / 建站以来」。

### 5.6 AI 增强策略（P0）

| 场景                  | Summary / 翻译 / 标签 / 质量分                             | Embedding（若用户开启） |
| --------------------- | ---------------------------------------------------------- | ----------------------- |
| 刷新增量（首次入库）  | 开关开 → 自动                                              | 开关开 → 自动           |
| 新订最新 5 条未读     | 开关开 → 自动                                              | 开关开 → 自动           |
| 迁移 / 手动补入的历史 | ❌                                                         | ❌                      |
| 库内其它已读存量      | ❌ 不自动                                                  | ❌ 不自动 backfill      |
| 打开文章              | **按钮**触发生成 summary（及实现时挂在同按钮上的单篇能力） | 不因打开强制            |

原则：**token 跟着注意力与显式操作走，不跟着库容量走。**

#### Embedding「存量」说明（避免误解）

- **Embedding** = 条目的向量表示，用于语义相近、Recommended 兴趣匹配、后续 topic 聚类；通常走 embedding 模型（如 bge-m3），与写 summary 的 LLM 不是同一类成本。
- **存量 embedding** = 库里已有原文、但还没有向量的老条目，要不要后台批量补算。
- **P0 决策：** 不补存量；只对「新入库 / 未读窗口」在开启 embedding 时自动算。等搜索 / 语义能力立项再单独定 backfill 策略。

### 5.7 打开文章与 Summary

- 打开条目 **不** 自动 enqueue summary
- 阅读区提供明确按钮（已有则展示，缺失则「生成摘要」类动作）
- 点击后仅针对 **当前 entryId** 跑 enrichment，范围可见、成本可预期
- 若用户在设置里关闭了 summary 能力，按钮应禁用或隐藏并说明原因（实现时对齐现有 AI 设置）

说明：条目若在「新入库自动流水线」里已经生成过 summary，打开时直接展示，无需再点。按钮针对的是 **尚未增强的存量 / 历史**。

### 5.8 存储与上限

- P0：**不对单次 ingest 设条数 soft cap**（feed 返回多少进多少）
- 长期仍可能需要全局 / 每源保留与淘汰策略（已读优先淘汰、收藏与 topic 引用保护）；**本 P0 不强制做淘汰产品化**，可在实现时保留扩展点
- 不把「无 cap」理解成「爬取站点全站历史」——边界仍是 **单次 RSS 响应窗口**

## 6. 对用户的影响（验收直觉）

| 之前                   | 之后                                            |
| ---------------------- | ----------------------------------------------- |
| 新订本地大约只有 5 条  | 新订 = feed 当前返回的都会进库，只有 5 条未读   |
| 老源永远偏瘦           | 升级后静默补满当前窗口（已读、不耗 AI）         |
| 担心补历史烧 token     | 迁移 / 导入默认 0 LLM；摘要要点按钮             |
| 时间线被历史淹没       | 未读仍约 5 条                                   |
| 以为「多刷新历史会来」 | 产品行为明确：增量靠刷新，窗口内历史靠迁移/导入 |

### 7.2 API 形态（建议）

```ts
// 概念接口，实现时可合并或改名
type IngestOptions = {
  /** 哪些 id 保持 unread；其余入库为 read */
  unreadEntryIds?: Set<string>
  /** 是否对 unread/新条目触发默认 AI；历史补齐应为 false */
  triggerEnrichment?: boolean
  /** 标记来源：initial | refresh | historyBackfill */
  reason: "initial" | "refresh" | "historyBackfill"
}

// 新订：全量 entries + unread = 最新 5 + enrichment 仅 unread
// 迁移/导入：全量缺失 entries + 全 read + enrichment off
```

常量语义调整：

```ts
// 语义从「只保留 N 条入库」改为「新订时保持 N 条未读」
export const INITIAL_SUBSCRIPTION_UNREAD_COUNT = 5
```

### 7.3 迁移 flag（建议）

- 全局：`local-rss:history-backfill-v1` = `done` | `pending` | 进度元数据
- 或每源：`local-rss:history-backfill-v1:${feedId}`
- 需可强制清 flag 以便开发与「手动全部重试」（若做设置项）

## 11. 成功标准（P0）

1. **语料厚度**：新订与迁移后，每源本地条数显著高于「仅 5 条」，并接近 feed 窗口大小
2. **时间线健康**：新订未读仍约 5，无历史洪水
3. **Token 健康**：迁移 / 导入不触发历史全量 LLM；摘要仅按钮或新内容自动流水线
4. **老源公平**：升级后老源与新源在「窗口内可获取条目」上可对齐
5. **预期清晰**：用户不会误以为刷新能拉回窗口外归档

---

## 12. 最短复述

1. **先厚订阅库，再做搜索。**
2. **入库放宽（全窗口）、未读仍紧（5）、AI 仍紧（新+未读；摘要可按钮）。**
3. **老源升级静默补窗口内历史，已读、不耗 AI。**
4. **RSS 只保证增量与当前窗口；窗口外不承诺。**
